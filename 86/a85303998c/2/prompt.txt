# Plan: Dual-Language (EN/ZH) Content Storage and Display

## Context
The current language implementation tells the AI to respond in a single language based on the toggle. The user wants **both EN and ZH versions stored for all content** — every conversation message and every frame section. When the language toggle is switched, the entire UI displays the corresponding version. Each message exchange produces both language versions via a post-response translation step.

---

## Approach

After the AI produces its primary response, make a **separate translation AI call** to translate both the user's message and the AI response to the other language. Store both versions on every message. Same approach for frame synthesis — synthesize first, then translate all sections. Translation failure is non-fatal (log + proceed with primary only).

Backward compatible: all new fields are Optional, display falls back to `content` when bilingual fields are missing.

## 2. Backend Translation Logic

### `src/backend/app/agents/conversation.py`

**Add `TRANSLATE_PROMPT`** — batch translate a JSON dict of key→text pairs, preserve markdown.

**Add `translate_texts()` method:**
```
async def translate_texts(self, texts: dict[str, str], source_lang: str, target_lang: str) -> dict[str, str]
```
- Single AI call, JSON response format
- Returns same keys with translated values

**Modify `process_turn()`** — after existing AI call:
1. Determine `primary_lang` = `language or "en"`, `other_lang` = opposite
2. Call `translate_texts({"user_message": user_msg, "ai_response": response}, primary_lang, other_lang)`
3. Set `user_content_en/zh` and `response_en/zh` on ConversationTurn
4. try/except: translation failure logs warning, continues without bilingual fields

**Modify `process_review_turn()`** — same pattern after the primary text call.

**Modify `synthesize_frame()`** — after primary synthesis:
1. Translate all non-empty sections to other language
2. Return dict with 15 keys: 5 primary + 5 `_en` + 5 `_zh`
